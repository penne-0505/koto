# 日本語プログラミング言語　設計メモ

## 1. プロジェクトの動機と目的

### 主目的
日本語ドリブンでプログラムを書くと、思考フレームがどう変わるかを観察する。

### 従目的
簡単なAPIが書ける程度の実用性を持たせる。「これですべてを書く」言語にはしない。

### 動機の背景
- 英語を知らないとプログラムが理解できないことへの違和感
- キーワードだけでなく、識別子（変数名・関数名）まで英語になることへの不満
- 言語は思考のフレームワークに強く作用するという信念（Sapir-Whorf的な視点）
- 語彙置換ではなく、日本語の文法をなるべく保った形にしたい

### なでしことの差異
なでしこは「日本語で書ける実用言語」が主目的。思考フレームへの作用は意図された主題ではない。
この言語は逆の重み付けで、思考への作用が主、実用性が従。

### 観察の手段
当面は自分で書いて内省する。他人に書かせる場合はドキュメントや学習教材が必要になるが、現時点では自己観察が基本。

---

## 2. 評価基準

~~「散文的であること」~~

→ **「日本語話者が読めて、かつ日本語の文法的特性が活きている構文か」**

散文であることが基準ではなく、以下の条件を満たすかどうかで構文を評価する。

- 日本語話者にとって読める
- 助詞が意味を運んでいる
- 語順自由性がある
- 日本語の構造を活かしている

## 3. 確定した設計方針

### 3-1. 語順と語彙

| 項目 | 方針 |
|------|------|
| 語順 | 自由。助詞が意味役割を運ぶので、語順を入れ替えても同じ意味として受理する |
| 語彙 | 固定。`数値配列` を `数値リスト` と書き換えることはできない。`繰り返す` を `ループする` と書けないのと同じ |
| 分かち書き | **全角スペース U+3000 が唯一の token 区切り**。newline と indent は purely visual で、token を区切る semantic role を持たない。multi-line でも token 境界には全角スペースを必ず維持する (trailing 全角 space を含む)。これにより単一行 ⇔ 多行の変換が機械的・無損失 |
| インデント | **tab 文字を canonical 採用**。視覚レイアウトのみ、semantic な役割なし。半角スペース不採用 |
| 数値 | 算用数字に統一。漢数字は使わない（熟語の構成要素は除く） |

#### 句読点ルール (Phase A 確定)

- `、` = 継続 (次に何か続く)
- `。` = 完結
- 動詞は対応する活用形を取る:
  - `、` の前は連用形 (`受け取り`、`して`、`なら`、`繰り返すと`、`をして待つ`)
  - `。` の前は終止形 (`受け取る`、`返す`、`例外として終える`、`である`、`保存する`)
- `である` は活用形不変で 2 役を担う: 型注釈マーカー (`〜 である 〜`) と節終結 copula (`である。`)
- `、` の細分 (列挙区切り vs 構造的遷移) は spec で分けない。書き手の感覚に任せる

#### compound keyword は内部 space なし (Phase B 確定)

particle + verb が一塊で機能する koto の predicate / control-flow 関連 keyword は、内部に space を入れない (compound keyword として扱う)。例:

`として取り込む` / `を受け取り、` / `を返す` / `として終える` / `で終える` / `について` / `について繰り返すと` / `何もしない` / `と同じ` / `より大きい` / `のいずれか` / `に含まれる` / `である`

「助詞で意識させる」目的は koto 全体の token 間 space (`X が Y` 形式) で既に実現されており、compound keyword 内の space は redundant。

### 3-2. 宣言の構文

宣言の種類を `〜として` で統一的に明示する。

```koto
関数　として　〜
オブジェクト　として　〜
定数　として　〜
可変　として　〜
```

#### 関数シグネチャの正式形 (Phase A 確定)

関数宣言は **「ヘッダ節」として 1 つの日本語のまとまり** を構成する:

```koto
[修飾] 関数 として [パラメータ列挙] [戻り値型] を返す [関数名] は、
    [本体]
である。
```

- **複数パラメータ** は `〜 を受け取り、` で連結する (連用形 + 、)
- **戻り値型は必須**。void なら `無し を 返す` を明示する
- 単一行版でも多行版でも書ける (token 間 space を維持する限り意味は同じ)

例 (多行):

```koto
公開　関数　として
　　数値配列　である　対象ら　を受け取り、
　　数値　を返す
最大値　は、
　　〜
である。
```

例 (単一行):

```koto
公開　関数　として　数値配列　である　対象ら　を受け取り、　数値　を返す　最大値　は、　〜　である。
```

#### 関数本体の終端 / 早期終了 (Phase B 確定)

- `である。` が正式 (言語史的に唯一の終端だった)
- `〜 を 返す。` (`無し を 返す。` 含む) も許容
- 本体が継続節 (〜、) で終わるなら `である。` で閉じる、return で終わるなら return の `。` で完結

**早期終了**:

- **戻り値なしの関数** (`無し を 返す` 関数): mid-body `である。` で関数を即終了させる。「ここで関数本体が終わる」という意味論を mid-body にも適用
- **戻り値ありの関数**: mid-body `〜 を 返す。` で値付き早期 return
- ~~`ここまで。` のような専用 early-exit keyword~~ → 不採用 (`である。` で十分、keyword 重複回避、vocabulary 固定原則)

### 3-3. 型の表現

- 型注釈は `〜である〜` で挟む：`数値 である 現在の最大`
- TypeScript的な型推論：推論可能なら省略できる、推論できない場面では必須
- 関数シグネチャの型（引数型・戻り値型）は常に明示

#### Optional 型: `T？` postfix (Phase A 確定)

~~「たぶん〜」候補~~ → 撤回。**`T？` postfix Optional を採用** (`？` は全角 U+FF1F)。

```koto
文字列？　である　メッセージ                ← Optional<string>
数値配列？　である　結果ら                  ← Optional<Array<number>>
タスク？　である　現在のタスク               ← Optional<Task>
```

#### `無し` の semantics (Phase A 確定)

**`無し` = unit 型の唯一の値**。void return (`無し を 返す`) も Optional の「無いほう」も同じ `無し` を使う。型システムが文脈を強制する。

- **非 Optional 値の null 比較は型エラー**: `T である X` に対して `X が 無し と同じ` を書くと型システムが拒否する
- **flow-sensitive type narrowing** (escape pattern のみ確定):
  ```koto
  もし　X　が　無し　と同じ　なら、
  　　「X が 無し」　を　例外　として終える。
  # ここから先、X は T？ から T に narrow される
  ```
- 「肯定 narrowing パターン」(条件式内で narrow する形) は未確定 (§5 参照)
- `T？？` のような二重 Optional は禁止 (無意味、型チェッカーで拒否)
- `？` は **Optional 専用記号**。述語関数名 (`〜か`、`空である` 等) には使わない (`存在する？` のような identifier suffix は不可)

型修飾子の詳細な合成規則は §3-23 を参照。

### 3-4. 集まり系の語彙

型名は複合名詞で書く。

| 概念 | 書き方 | 空のもの |
|------|--------|----------|
| 配列 | `数値配列`、`タスク配列` | `空配列` |
| 集合 | `数値集合`、`タスク集合` | `空集合` |
| 辞書 | `文字列とタスクの辞書` | `空辞書` |
| 組 | 保留（知名度低、PoCでは不要かもしれない） | — |

辞書は二項を取るため、複合名詞ではなく `〜と〜の辞書` で表現する。

### 3-5. インデックス / indexed accessor (Phase B 確定)

1起点。**inclusive 4 形** を採用する:

| form                          | 意味                       |
|-------------------------------|----------------------------|
| `〜 の N 番目`               | 単一要素 (1-indexed)        |
| `〜 の N 番目以降`           | N 以降全部                  |
| `〜 の N 番目以前`           | 1 から N まで (inclusive)   |
| `〜 の N 番目から M 番目まで` | N から M まで (inclusive)   |

- **exclusive 形** (~~`より後`~~ / ~~`より前`~~) は不採用 (比較演算で `以上`/`以下` のみ採用し ~~`未満`~~ / ~~`超える`~~ を不採用とした方針と整合)
- exclusive を書きたい場合は operand を ±1 して inclusive 形に書き換える (`5 番目より後` → `6 番目以降`)
- `〜番目` の前は算用数字に限定 (`一番目` は不可)
- **戻り値型**:
  - 単一要素 `N 番目`: 元要素の型 + `？` (Optional) — out-of-bounds は `無し`
  - 範囲アクセサ: 元の collection 型 (配列 → 配列、文字列 → 文字列)。out-of-bounds は空 collection
- **適用範囲**: 文字列・配列 の built-in 機能。順序のない collection (辞書 / 集合 等) では別 access 方法

### 3-6. ミュータビリティ

- デフォルトは不変
- 可変にしたい場合は `可変として` を明示
- `定数として` は明示してもいいが、普通は不要（デフォルトが不変なので）

```koto
タスク配列　である　結果　を　空配列　とする。               ← 不変（デフォルト）
可変として　タスク配列　である　結果　を　空配列　とする。    ← 可変
定数として　タスク配列　である　結果　を　空配列　とする。    ← 不変（明示、省略可能）
```

### 3-7. 等価判定と比較演算 (Phase B 拡張)

すべて言語組み込み (built-in keyword) として実装。user 定義不可。

#### 等価判定

| 構文 | 意味 | 英語圏での対応 |
|------|------|----------------|
| `〜 と同じ` | 値として等しい | `==` |
| `〜 と異なる` | 値として等しくない | `!=` |
| `〜 そのもの` | 同一インスタンス | `is` |
| `〜 と同種` | 同一型 | `type(a) == type(b)` |

#### 順序比較 (Phase B 追加)

| 構文 | 意味 |
|------|------|
| `〜 より大きい` | > |
| `〜 より小さい` | < |
| `〜 以上` | ≥ |
| `〜 以下` | ≤ |

~~`未満`~~ / ~~`超える`~~ → 不採用 (`以上`/`以下` の operand を ±1 して書く、vocabulary 固定原則)

#### 真偽値の比較は明示 (Phase B 確定)

- `もし X が 真 と同じ なら、〜`
- `もし X が 偽 と同じ なら、〜`
- ~~暗黙の `もし X なら、〜`~~ → 不採用

#### 論理結合 (Phase B 確定)

| 構文 | 意味 | 備考 |
|------|------|------|
| `〜 かつ 〜` | AND | |
| `〜 または 〜` | OR | boolean 専用、value 列挙には §3-21 を使う |
| `〜 ではない` | NOT | 条件式末尾、`である` の対の copula として一塊で扱う |

~~`もしくは`~~ / ~~`なおかつ`~~ → 不採用 (canonical の `または` / `かつ` を使う、vocabulary 固定原則)

#### 優先順位 (Phase B 確定)

優先順位 (高→低):

1. 比較演算 (`と同じ` / `と異なる` / `と同種` / `そのもの` / `より大きい` / `より小さい` / `以上` / `以下`)
2. `ではない` (否定。直前の単一比較式に attach)
3. `かつ` (AND)
4. `または` (OR)

同レベル内は **左結合** (左から右に評価)。

- `A かつ B または C` → `(A かつ B) または C`
- `A かつ B かつ C` → `(A かつ B) かつ C`
- `A かつ B ではない` → `A かつ (B ではない)`
- `ではない` は単一比較に attach、複合全体への否定は不採用 (De Morgan で書き換えるか節分割)

#### grouping 不採用 (条件式の括弧)

`（A かつ B）または C` のような括弧グルーピングは **採用しない**。

理由:
- 自然日本語の話し言葉に grouping 記号がない (書き言葉の技術文書には存在するが、「読める」レベルで「流れるように読める」ではない。koto は後者を目指す)
- ネストするほど複雑な条件式はそもそも書くべきではない (書き手の意図が曖昧)
- 中間概念に名前を付けて節分割するか、De Morgan で書き換えるほうが Japanese flow に乗る

複雑な条件が必要な場合は **節分割** または **中間真偽値変数** で書く。

### 3-8. 関数呼び出し

全角丸括弧 `（）` で引数を囲む。複数引数は読点で区切る。

```koto
最大値（対象ら）
未完了の取り出し（タスクら）
何かの処理（引数1、引数2、引数3）
現在の日時（）          ← 引数なしでも括弧は必須
```

#### 位置引数とキーワード引数 (Phase A 確定)

両対応 (関数呼び出し・コンストラクタ呼び出しで共通形):

- 位置引数: `関数名（V1、V2）`
- キーワード引数: `関数名（フィールド名 が V1、別フィールド が V2）`
- 混在 (位置 → キーワード) も可

### 3-9. 制御フロー

#### return

`〜 を 返す。` で関数を完了する。「返す」は完了感を内包する。

#### 異常終了 (throw)

§3-19 へ分離。`〜 を [終了型] として終える。` / `[終了型(...)] で終える。` の両立形で記述する。

#### 条件分岐

`もし 〜 なら、〜。` で表現。if-else は強制しない。

```koto
もし　今の数　が　現在の最大　より大きい　なら、
　　現在の最大　を　今の数　とする。
```

#### 繰り返し

`要素を 〜 として 〜 の 各要素について繰り返すと、〜 である。`

```koto
要素を　今の数　として　対象ら　の　各要素について繰り返すと、
　　（本体）
である。
```

ループ構文の統一 (`〜 について繰り返すと、` 形 と `〜 の 各要素について繰り返すと、` 形の整理) は §5 で保留。

#### match 構文

§3-22 へ。`〜 について、 [case なら、 [body]]` 形のディスパッチ構文。

#### no-op

§3-24 へ。`何もしない。` で「何もしない」を明示。

### 3-10. 記号の用途 (Phase B 拡張)

| 記号 | 用途 |
|------|------|
| `「」` | 文字列リテラル |
| `（）` | 関数呼び出し・コンストラクタ呼び出しの引数 |
| 句点 `。` | 文の終端。ブロックを閉じる役割も兼ねる |
| 読点 `、` | ブロックの開始、列挙の区切り (継続マーカー) |
| 中点 `・` | 集合 disjunction の値列挙 (`V1・V2・V3 のいずれか`) — §3-21 |
| `？` | 型 postfix Optional (`T？` = `Optional<T>`) — §3-3 / §3-23 |
| 全角スペース U+3000 | 分かち書き = 唯一の token separator (§3-1) |
| tab | indent (視覚レイアウトのみ、semantic な役割なし、§3-1) |

識別子は裸で書く（括弧なし）。

### 3-11. 型定義

#### オブジェクト

```koto
オブジェクト　として　タスク　は、
　　文字列　である　タイトル、
　　タスク状態　である　状態、
　　日付　である　締め切り、
からなる。
```

#### 列挙型

```koto
タスク状態　は、未完了、進行中、完了　のいずれか。
```

列挙値の参照時、フィールドの型から推論できる場合は型名を省略可能。
`今のタスク の 状態 が 未完了 と同じなら` と書ける（`タスク状態の未完了` と書かなくてよい）。

#### インスタンス生成

```koto
タスク（
　　タイトル　が　「買い物」、
　　状態　が　未完了、
　　締め切り　が　今日
）　を　新しいタスク　とする。
```

フィールドの順序は自由 (助詞でフィールド名を指定しているため)。
型が自明な場合は型注釈を省略可能。

#### メソッド: 不採用 (Phase B 確定)

~~メソッド (将来追加)~~ → **不採用確定** (~~`関数 として タスク の 完了する は、〜`~~ も ~~`自身`~~ も撤回)。

理由:
- メソッド呼び出しは日本語の動詞-粒子構造と衝突する
- `obj.method(arg)` 形が OOP 的思考を引っ張る
- 自由関数で subject を明示的に引数として渡す方が日本語の flow に乗る

「method = instance への bound dispatch」を狭く定義し、それのみを禁止する。type-as-namespace (`型 の 関数(...)` の関連 free function 公開) は許容 (§3-25 参照)。

タスクへの操作は自由関数として書く:

```koto
関数　として
　　タスク　である　対象　を受け取り、
　　無し　を返す
タスク完了　は、
　　対象　の　状態　を　完了　とする。
である。
```

### 3-13. モジュールシステム (Phase B 拡張)

**ファイル＝モジュール。** 一つのファイルが一つのモジュールに対応する。

#### 取り込み (import)

```koto
タスク定義　から　タスク　を　取り込む。
タスク定義　から　タスク、タスク状態　を　取り込む。
タスク定義　を　タスク補助　として取り込む。       ← module alias
```

- 個別名 alias (`〜 から X を Y として 取り込む。`) は未確定 (§5 参照)
- ~~`〜 として 全て取り込む`~~ → 撤回 (module alias で必要を満たすため)

#### 可視性

デフォルト非公開。公開したいものは `公開` を宣言の頭に付ける。

```koto
公開　関数　として　数値配列　である　対象ら　を受け取り、　数値　を返す　最大値　は、
公開　オブジェクト　として　タスク　は、
```

`公開` は分かち書きで前置する (複合名詞にしない)。修飾語が複数ある場合に積み重ねられるようにするため。

```koto
公開　可変として　タスク配列　である　結果　を　空配列　とする。
```

#### モジュールの参照方法: パスリテラル

鉤括弧 `「」` の中にファイルパスを書く。

```koto
「タスク定義」から　タスク　を　取り込む。                          ← 同じ階層
「モデル/タスク定義」から　タスク、タスク状態　を　取り込む。      ← 下位階層
「../ユーティリティ/日付変換」から　日付変換　を　取り込む。        ← 相対参照
```

書いたパスとファイルシステム上の実体が一致する。拡張子は省略。

#### `の` chain 経由のアクセス

`の` chain で module 内のものにアクセスできる。深さに制限あり (詳細は §3-25):

- module alias で取り込んだ場合: `Y の 関数(...)` (1 段の namespace lookup)
- 取り込まずに直接アクセスはできない (必ず import を経由)
- 深く navigate したい場合は import を深くする (`「../module/sub」 を Sub として 取り込む。`)

### 3-14. トランスパイル先とFFI

**トランスパイル先：JavaScript（Node.js）。**

あなたの言語のソースコードをJavaScriptファイルに変換する。変換されたコードはJavaScriptそのものなので、JavaScriptの全ライブラリが使える。将来フロントエンドに進む道も開ける。

**外部ライブラリの呼び出し：宣言ファイル方式（TypeScriptの `.d.ts` に相当）。**

外部JavaScriptライブラリを使うとき、あなたの言語の構文で「外の世界にこれがある」と宣言する。利用者はJavaScriptを直接書く必要がない。コード生成器が宣言を見て適切なJavaScriptを出力する。

```koto
外部　オブジェクト　として　クライアント　は、
　　文字列　である　トークン　を受け取り、　無し　を返す　ログイン、
　　からなる。
```

宣言ファイル自体もあなたの言語の構文で書く。型チェックが効く。

**標準ライブラリ：** JavaScriptの既存ライブラリの日本語ラッパーとして段階的に構築する。APIデザイン（関数名、助詞パターン、型の使い方）はこの言語の設計原則に従い、利用者はJavaScriptの概念を意識しない。

### 3-16. プロパティアクセスと添字アクセス (Phase B 再構成)

いずれも `〜 の 〜` で書く。

```koto
対象ら　の　1番目                ← 配列の添字アクセス (§3-5、indexed accessor)
対象ら　の　要素数               ← built-in property (後述)
タスク表　の　「タイトル1」     ← 辞書のキーアクセス
今のタスク　の　状態             ← オブジェクトのフィールドアクセス
```

添字とキーの区別は型情報で解決。構文上は区別しない。

#### property access (built-in 型のみ)

immutable binding 上で field-syntax 経由の派生値アクセスを許可する。「state はフィールド、派生はファンクション」がデフォルト原則、ただし **immutable な値の derived は constant と区別できない** ので field 化する。

- **適用範囲: built-in 型のみ**。user 定義型は宣言された field のみアクセス可能、派生値は free function を使う
- **binding mutability 制約**: property syntax は immutable binding 上でのみ使える。`可変 として` で宣言された binding には適用されず、free function `関数名(値)` を使う
  - 理由: mutable binding 上で `value の derived` を許すと、同一式内で derived 値が変化する surprise が起こる (`X の 長さ + X の 長さ` で X が変わると 2 値が異なる)。immutable に限定することで構造的に防ぐ
- **意味論**: 概念的には「定義時に派生値を計算してフィールドとして保持」。実装は lazy + memoize でも eager pre-compute でも、observable には同じ
- **メソッド不採用との関係**: property は parameterless / pure / 名詞句で読める derived value。implicit self を取って args を持って dispatch される method とは別物。`の` chain の自然な拡張として扱う

#### inclusion criteria

built-in property を増やすときの目安:

1. 常に定義される (failure mode なし、Optional 返さない)
2. 計算が conceptually trivial (metadata 的、transformation ではない)
3. noun として読める (`〜 の 長さ` は noun、`〜 を 大文字化` は verb)
4. 戻り値型が原値より simple

`空である` / `先頭文字` / `最後の要素` 等の failure mode を持つもの・predicate 系は property に含めず、predicate function / Optional 返り関数として実装。

#### built-in 型ごとの property 一覧 (初版 draft)

| 型       | property | 戻り値型 |
|----------|----------|----------|
| 文字列   | 長さ     | 数値     |
| 配列     | 要素数   | 数値     |
| 日付     | 年       | 数値     |
| 日付     | 月       | 数値     |
| 日付     | 日       | 数値     |
| 日付     | 曜日     | 列挙型   |
| 時刻     | 時       | 数値     |
| 時刻     | 分       | 数値     |
| 時刻     | 秒       | 数値     |
| 日時     | 日付部   | 日付     |
| 日時     | 時刻部   | 時刻     |

以降は実コードを書きながら追加。

#### mutable / 派生 / user 定義型での代替手段

`要素数(配列)` / `長さ(文字列)` のような free function を使う。built-in 型でも mutable binding なら同じ。

#### vocabulary 補足

string は `長さ`、array は `要素数` と分ける (vocabulary 固定原則、混同を避ける)。

### 3-17. 非同期処理

**非同期関数の宣言：** `非同期` を修飾語として前置する。

```koto
非同期　関数　として
　　メッセージ　である　メッセージ　を受け取り、
　　無し　を返す
メッセージ処理　は、
　　もし　メッセージ　の　内容　が　「!ping」　と同じ　なら、
　　　　メッセージ　に　「pong」　をして待つ。
である。
```

`公開 非同期 関数 として` のように修飾語を積める。

**await：** 日本語のて形接続を使う。`〜して待つ`。

```koto
メッセージ　に　「pong」　を返信して待つ。
データ　を　取得して待つ。
```

「返信する」＋「待つ」をて形で繋ぐ。日本語として自然に読める。

### 3-18. 高階関数とmap/filter

**関数の受け渡し：** 名前付き関数の名前をそのまま値として渡す。特別な構文不要。

```koto
クライアント　の　イベント登録（「メッセージ受信」、メッセージ処理）。
ボタン　の　クリック時に（処理）。
```

イベント登録は言語構文ではなくメソッド呼び出し ~~（）~~ → 関数呼び出し (メソッド不採用に伴う書き換え)。ライブラリのAPIデザインの問題。

**匿名関数：PoCでは不要。** 名前付き関数の受け渡しで対応する。「匿名関数がないとき、書き手は何に名前を付けることになるか」は思考フレームへの作用の観察対象。必要と感じたら後から追加。

**map/filter：標準ライブラリとして実装。** 言語構文ではなく配列に対する関数。宣言的な書き方を採用予定。

```koto
対象ら　の　各要素　の　タイトル　を　集める。          ← map相当
対象ら　の　うち、状態　が　未完了　と同じ　であるもの。  ← filter相当
```

宣言形の正式構文化は Phase B 持ち越し (§5 参照)。

### 3-19. 異常終了 / 終了型 (Phase B 新規)

#### 2 つの出口モデル

関数は **正常終了の出口 (`〜 を 返す。`)** と **異常終了の出口 (`〜 で終える。` / `〜 として終える。`)** の 2 つを持つ。宣言部の `〜 を 返す` は **正常終了の出口だけを縛る**。異常終了は別 channel として独立。

#### 終了型

異常終了で投げる値は **「終了型」と分類された型** でなければならない。

- **言語組み込み 終了型 (初版)**: `例外`、`警告`、`通知` (詳細仕様は今後)
- **user 定義 終了型**: 何らかの marker で 終了型 として登録できる構文を将来詰める。built-in 3 つで実用の多くはカバー

#### 2 つの書き方を許容 (両立形)

**explicit 形**:

```koto
例外（メッセージ　が　「メッセージが無し」、　コード　が　〜）　で終える。
```

- `例外（…）` は普通の関数呼び出し (引数を明示) → `で終える` keyword に渡して異常終了
- 何が構築されているか **書かれた行から見える** (暗黙性なし)
- 複雑な引数を取る場合に推奨

**sugar 形**:

```koto
「メッセージが無し」　を　例外　として終える。
```

- explicit 形の短縮形。`〜 を Type として終える。` は `Type（〜） で終える。` と等価 (Type は終了型名のプレースホルダ)
- 終了型 が **単一引数 (典型的に メッセージ) のみで構築可能** な場合のみ使える
- 軽量な場面で推奨

#### 型制約

- `で終える` の引数は **終了型 として分類された型** でなければならない
- ~~`「文字列」 で終える`~~ のような無印値を直接投げる書き方は **型エラー**
- `として終える` の sugar 適用条件は仕様で限定 (詳細は §5 で正式化)

#### 終了型同士の親子関係 (継承) は不採用

`例外` `警告` `通知` 等は **横並びの独立した型** として並べる。継承による subtype catch は使わず、「どれかに当てはまる」を書きたい場合は §3-21 の集合 disjunction で `例外 ・ 警告 のいずれか として 捕まえる` 形で対応する。

#### デザイン哲学

「メソッド不採用 + 自由関数中心」と整合的。`例外(...)` は自由関数の呼び出し、`で終える` は独立した control-flow keyword。両者を **構築と終了の 2 道具に分離** することで、行頭で何が起きているか読み解ける。

### 3-20. predicate と comparison の構文と実装方針 (Phase B 新規)

#### 比較演算 (comparison) は built-in keyword

§3-7 で列挙した `と同じ` / `と異なる` / `と同種` / `そのもの` / `より大きい` / `より小さい` / `以上` / `以下` は **言語組み込みの built-in keyword**。

- 他言語の `=` / `>` / `<=` 等 operator に相当する基礎概念
- 言語組み込みの方が一貫性と効率の両面で適切
- user 定義は不可

#### predicate function は default function

`で始まる` / `で終わる` / `を含む` / `に含まれる` / `のいずれか` 等は **標準ライブラリ / user 定義のいずれも default function 形**。

- **関数名に particle (`が` / `で` / `を` / `と` 等) を埋め込む命名規約**
- natural-language 呼び出し形: `[subject] [particle] [arg] [predicate-name]` が `predicate-name(subject, arg)` の syntactic sugar として parse される (Smalltalk keyword message / ObjC selector に類似)
- 既存「メソッド不採用」「free function 中心」原則と整合
- user は独自の predicate を同じ枠で定義可能

例:

```koto
もし　X　が　「!」　で始まる　なら、     ← で始まる は predicate function
```

これは `〜が〜で始まる(X, 「!」)` として parser が解釈する。

#### 副次論点 (今後詰める)

- particle 埋め込み命名規約の formalization (declaration syntax の詳細、許される particle の種類)
- predicate form が許される条件 (arg 数上限、戻り値型、命名形式の制約)
- paren 形 `関数名(P が V, ...)` と predicate form `[subj] [part] [arg] [name]` の使い分け方針

詳細は §5 参照。

### 3-21. 集合 disjunction (Phase B 新規)

#### value 位置の `または` は不採用

§3-7 で確定した `または` は **boolean レベルの結合専用**。value 位置で disjunction を表現したい場合は、本セクションの集合構文を使う。

#### 修飾形 (predicate の引数位置)

`LHS が V1・V2・... のいずれか [predicate]`

```koto
もし　X　が　「!」・「！」　のいずれか　で始まる　なら、
```

`〜 のいずれか` は集合 disjunction を構築する修飾形。

#### predicate 完結形 (集合メンバシップそのもの)

`LHS が V1・V2・... に 含まれる`

```koto
もし　X　が　「ping」・「pong」・「time」　に　含まれる　なら、
```

集合メンバシップを述語として完結。

#### 列挙記号: 中点 `・` (U+30FB)

値の列挙には中点を使う。`、` は continuation 記号として温存し、value 列挙には使わない。日本語の伝統的な列挙記号 (人名・カテゴリ・選択肢) として自然。

#### negation との合成は構造的に明確

`〜 のいずれか で predicate ではない` / `〜 に 含まれる ではない` は「集合の要素すべてで predicate が成立しない」と一意に解釈される。De Morgan を読み手に強いない。

```koto
もし　X　が　「!」・「！」　のいずれか　で始まる　ではない　なら、
# → 「! でも ！ でも始まらない」と一意
```

#### 値が 1 つの場合

通常の predicate (`X が V で始まる`) で書く。`〜 のいずれか` / `〜 に 含まれる` は集合 (2 値以上) に対して使う。

#### 採用しなかった他案 (歴史)

- ~~value 位置で `または` を distribute する案~~: edge case (negation 合成での De Morgan 必要) が構造的問題
- ~~冗長 boolean 形のみ許可~~: 書き味の負担が大きい、koto の「流れる日本語」goal と衝突
- ~~新記号 bracket (`〔〕` 等) 採用案~~: IME 入力負荷が高い

採用言語の前例: SQL `IN (...)`、Python `x in (...)`、Ruby `[...].include?(x)`、Rust pattern `v1 | v2`、Kotlin `when` の case 列挙等、主要モダン言語の (c) 系統への収束的選択。

### 3-22. match 構文 (`〜 について、`) (Phase B 新規)

#### 構文形

```text
[subject] について、 [case なら、 [body]] [case なら、 [body]] ...
```

switch / match 風のディスパッチ構文。

#### case の condition は subject に対する省略が許される

- `[X] と同じ なら、` → 暗黙に `subject が X と同じ なら、`
- `[Type] と同種 ではない なら、` → subject の type 判定
- **暗黙フィールドアクセス**: `[field名] が 〜 なら、` → `subject の [field名] が 〜 なら、`

#### 暗黙フィールドアクセスの conflict 解決

subject の field が優先。outer scope の同名識別子は隠される。外側変数を参照したい場合は明示 prefix で逃げる必要がある。

#### 適用範囲

match 構文専用。汎用 scope opener (`について、` を任意 block に展開する) は将来検討。

#### case body と終端

- **case body**: 1 文以上書ける。次の `[condition] なら、` 行で前 case が暗黙に閉じる
- **終端**: 暗黙終端。次の clause が `[condition] なら、` 形式でなければ match 全体が終わる。明示終端 keyword なし
- **default**: 不要、暗黙 noop。書きたければ最後に `その他 なら、〜` を書く (escape valve)。型システムによる exhaustiveness 強制は行わない
- **fall-through なし**: matched case 実行後 match を抜ける。複数 case を同じ body で受けたい場合は condition の集合 disjunction (§3-21) で対応

#### デザイン哲学

「`その他 なら、〜` を書きたくなったら設計を見直す」という signal を意図。網羅性の自動チェックではなく、書き手の判断に委ねる方針 (§1「思考フレームへの作用」主目的との整合)。

#### parser 着手時の caveat

- ネスト match のスコープ追跡 (case body 内の別 match の case を外側 match の case と誤読しない手当て)
- `もし X なら、` (if-statement) と match case の `[condition] なら、` の構文的区別 (`もし` 接頭の有無で判定する案が候補)

### 3-23. 型の postfix 修飾子合成 (Phase B 新規)

#### 構文形

型修飾子は postfix で適用し、左→右に累積的に包む。

#### 修飾子 (現状確定)

- `？` (Optional): `T？` = `Optional<T>`
- `配列` (Array): `T配列` = `Array<T>`

#### 合成例

| 表記             | 意味                                  |
|------------------|---------------------------------------|
| `文字列`        | string                                |
| `文字列？`      | `Optional<string>`                      |
| `文字列？配列`  | `Array<Optional<string>>`               |
| `文字列配列？`  | `Optional<Array<string>>` (順序で意味が変わる) |
| `文字列配列？配列` | `Array<Optional<Array<string>>>` (3 段) |

#### practical guidance

- `配列？` (`Optional<Array>`): 「配列ごと存在しないかもしれない」 — 普通の use case (HTTP の optional クエリ、API の optional field 等)
- `要素？配列` (`Array<Optional<要素>>`): 「配列はあるが個別要素が欠けている可能性」 — 稀。配列要素 への index access (§3-5 indexed accessor) は自動で `Optional<T>` を返すため、明示的に `Array<Optional<T>>` を作る必要はほぼない。引数が無い場合は **長さ 0 の `文字列配列`** で表現する方が筋

#### 任意 depth で合成可能

修飾子を後置するだけ。書かれた順序が型の包み構造を決める。

#### 将来追加予定の修飾子

`辞書` / `集合` / `タプル` 等は登場時に追加 (§5 参照)。

### 3-24. no-op 構文 (Phase B 新規)

#### 構文形

`何もしない。`

Python の `pass` 相当の no-op statement。

#### 使う場面

if / match の body で「条件にマッチしたが何もせず通過させたい」とき。空 body よりも意図が明示される。

### 3-25. `の` chain の意味論 (Phase B 新規)

#### `の` の 3 つの役割

- field access (value → field)
- module navigation (module → member)
- type-as-namespace (type を関数や定数の置き場として使う)

構文形は共通の `LHS の RHS [の RHS ...]`、LHS の種類で意味が決まる。

#### 深さ制限

`の` chain は **module / type 境界を 1 回まで** しか跨げない。深く navigate したい場合は import を深くする。

- 例: `ディスコード の ギルド の 有効コマンド一覧(...)` ❌ → `「../ディスコード/ギルド」 を ギルド として 取り込む。` の上で `ギルド の 有効コマンド一覧(...)` ✅
- 理由: import を「状況設定」として明示化、call site を短く保つ、`の` chain の意味論を単純化、依存追跡を明確化

#### 意味論ルール (parse 後の type check で enforce)

- `value の field` → field access、OK
- `value の field の field の ...` → 多段 field 鎖、OK (data structure を辿るのは自然、深さ制限の対象外)
- `value の function(...)` → **instance method dispatch、禁止** (§3-11 「メソッド不採用」)
- `module の function(...) / module の type / module の constant` → namespace lookup (1 回)、OK
- `type の function(...) / type の constant` → associated free function / static constant (1 回)、OK (type を namespace として使う、Rust の `impl` 内 関連関数に近い)
- `module の module の thing` → namespace lookup 2 回、❌ → import を深くする
- `module の type の constant/function` → 同上、❌ → import を深くする
- `namespace の field の function(...)` → 最終的に value から function 呼び出し、method dispatch なので ❌

#### 判定表

| chain 形式                            | 判定 | 理由                                     |
|---------------------------------------|------|------------------------------------------|
| `value の field`                      | ✅   | field access                             |
| `value の field の field の ...`      | ✅   | data structure 鎖                        |
| `value の function(...)`              | ❌   | instance method dispatch                 |
| `module の function(...)`             | ✅   | namespace lookup (1 回)                  |
| `module の type`                      | ✅   | 同上                                     |
| `type の constant / function(...)`    | ✅   | 同上 (type-as-namespace)                 |
| `module の module の thing`           | ❌   | namespace lookup 2 回 → import を深く    |
| `module の type の constant`          | ❌   | 同上                                     |
| `namespace の field の function(...)` | ❌   | 最終 step が method dispatch             |

#### 「メソッド不採用」の正確な定義

instance に bound dispatch される形式 (`value の function(...)`) のみを禁止。type が namespace として free function を公開するのは OK (ただし chain 深さ制限内)。§3-11 の「メソッド不採用」を狭く解釈する形で確定。

#### namespace conflict 時の canonical 解法

`「..」 を Y として 取り込む。` で alias 取り込み → `Y の 関数(...)` で qualify する。TypeScript / Rust / Python / Go 共通パターン。既存 import 構文で十分カバー。

#### parse vs check の分業

parse 時は `LHS の RHS [...]` を構文的に許容、type check で LHS の種類 (value / module / type)、RHS の使われ方 (field 参照 / function 呼び出し)、namespace 境界を跨ぐ回数を検証して illegal を拒否する。

### 3-26. 変数束縛 canonical 形 (Phase B 新規)

#### canonical 形

```koto
[Type 修飾] [Type] である [変数名] を [値] とする。
```

`を` 形のみ canonical。

#### `は` 形は不採用

~~`A である B は C とする。`~~ → 不採用 (Phase B、A-8 で確定)

日本語的に自然な topic 形だが、vocabulary 固定原則と consistency 優先。既存の `〜 を 返す。` / `〜 を 受け取り、` / `〜 を Y として 取り込む。` 等と `を` で一貫。

#### 例

```koto
タスク配列　である　結果　を　空配列　とする。
可変として　数値　である　現在の最大　を　0　とする。
真偽値　である　成功した　を　偽　とする。
```

## 6. 設計上の原則（暗黙に合意されたもの）

- 「日本語プログラミング言語」ではなく「日本語の特性を活かしたプログラミング言語」
- 慣習に従う動機がない。慣習から外れることで「制御フローの言語化」が進む場面がある
- PoCではIR（中間表現）までを第一段階とし、JavaScript生成を第二段階とする
- 「なぜ作るか」は思考フレームへの作用の観察。実用ツールとしての生産性は二の次
- 英語の概念世界は輸入品であることを認める。日本語化の意味は「概念を日本由来にする」ことではなく「助詞・語順・修飾の自由度を日本語話者にとって自然にする」こと
- この言語にとってのエコシステムは「観察に十分な厚みのあるプログラムを書くために最低限何が要るか」から逆算する。一般的な言語の「ユーザーを集めて育てる」エコシステム論とは異なる
- JavaScriptにトランスパイルする。外部ライブラリは宣言ファイル方式で型付きで呼ぶ。標準ライブラリはJavaScriptライブラリの日本語ラッパーとして段階的に構築し、利用者はJavaScriptの概念を意識しない
- vocabulary 固定原則を全 spec で徹底する。canonical form を 1 つに絞り、不採用 form は撤回マーク付きで歴史に残す (Phase A/B で再確認)
